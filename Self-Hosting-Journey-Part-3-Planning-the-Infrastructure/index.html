Self-Hosting Journey - Part 3 - Planning the Infrastructure

Draka
2025-03-07
Self-Hosting
Architecture, Container, Docker, Infrastructure, Planning, Proxmox, Virtualization

Planning for Self-Hosting: Laying the Groundwork

While we wait for the hardware to be delivered, we can begin thinking about what we would like to host on our server and which technologies we want to use. This is the planning phase, a very important initial step that allows us to clarify our ideas and set a strong foundation for the successful deployment and operation of our server in the future.

Physical Architecture: Building the Foundation

As mentioned in the previous blog post, my setup includes a Mini-PC with three SSDs and 32GB of memory, enabling me to run multiple virtual machines (VMs) and containers simultaneously. To ensure data reliability, the SSDs are organized such that the main operating system (OS) resides on one disk, while the other two disks are configured as RAID1 for storing data and backups.

In addition to the Mini-PC, I have two other devices: a Raspberry Pi and an HP Elitedesk. The Raspberry Pi will host an offsite copy of my data, located at my parents’ house, which is 400 km away. This offsite backup will provide an additional layer of protection in case of a disaster, such as a fire or earthquake. The HP Elitedesk has 64GB of RAM but consumes more power than the Mini-PC and is not suitable for 24/7 operation. Therefore, it will only be powered on as needed, primarily serving as a host for a small research security lab.

Logical Architecture: Organizing Services and Workloads

With the hardware in place, it’s time to think about the logical structure of the system. How will we deploy and manage our services?

The most efficient and convenient way to host multiple services on a single machine is through virtualization. This is the same approach used by most cloud providers in their data centers, as it allows for flexible software deployment that is nearly independent of the underlying hardware.

Introduction to Virtualization

Virtualization is a technology that enables the creation of multiple virtual environments on a single physical machine. A hypervisor, or virtual machine monitor (VMM), manages these virtual machines (VMs), allowing each VM to run an operating system independent of the hypervisor itself. Most instructions are executed directly on the host’s hardware, providing a high degree of efficiency.

There are two types of hypervisors:

  • Type 1 (Bare-Metal Hypervisor): Runs directly on the physical hardware, offering better performance and security (e.g., VMware vSphere, Microsoft Hyper-V, Xen).
  • Type 2 (Hosted Hypervisor): Runs on top of an existing operating system, which introduces some overhead in terms of performance (e.g., VMware Workstation, VirtualBox).

In Linux, KVM (Kernel-based Virtual Machine) allows the system to function as a Type-1 hypervisor. It uses hardware virtualization features such as Intel VT-x or AMD-V and, when paired with QEMU (an open-source emulator), provides the ability to create and manage VMs.

Type 1 vs Type 2 Hypervisors

Another key technology is containerization, an operating-system-level virtualization that enables the creation of isolated user-space instances. Unlike VMs, containers share the same host OS kernel, which makes them lighter and faster but introduces potential security concerns due to their lower level of isolation. Docker, which rose to prominence in 2014, is the leading tool for deploying applications in containers, simplifying the management of dependencies and libraries.

Virtual Machines vs Containers

The following are the main advantages offered by the virtualization technologies

  • Isolation: Containers and VMs provide isolated environments for each service, preventing conflicts and ensuring security. This allows services to run independently without interfering with one another.
  • Resource Efficiency: Containers share the host OS kernel, making them lightweight and faster to start, while VMs provide full isolation with their own OS, offering more flexibility. Both methods allow better resource utilization.
  • Portability: Containers are highly portable, meaning services can run consistently across different environments (development, testing, production). VMs offer similar benefits with complete OS environments.
  • Scalability: Containers are easy to scale up or down for services based on demand, and VMs can scale to allocate more resources when needed.
  • Management: Both containers (with orchestration tools like Kubernetes) and VMs (with hypervisors) allow for easier management of multiple services, automating deployments, updates, and scaling.

While both VMs and containers offer advantages, they differ in key areas:

Feature Virtual Machines (VMs) Containers
Isolation Stronger isolation Weaker isolation
Resource Usage Higher overhead Lightweight
Security Better (full OS isolation) Lower (shared kernel)
Portability Less portable Highly portable
Performance Slower Faster

VMs vs Containers

Given my focus on control and security, I’ve decided to use a hybrid approach that combines both VMs and containers. This allows me to take advantage of the unique strengths of each technology while mitigating their weaknesses.

Operating System Selection: Building on the Right Foundation

For the operating system, I evaluated two options. The first is a basic Linux distribution (e.g., Debian) combined with libvirt for managing virtualization technologies like KVM, QEMU, and LXC containers. The second is Proxmox, a Debian-based platform that integrates both KVM for VMs and LXC for containers. Proxmox offers a user-friendly web interface for managing resources and includes features like live migration, distributed storage management (e.g., Ceph), and high availability through VM replication across nodes. Ultimately, I chose Proxmox for its versatility and ease of management.

The Hybrid Approach: VMs and Containers

I’ve decided to use Proxmox as the base OS for managing my VMs and LXC containers, while also leveraging Docker to simplify the deployment of certain applications. The three main options for integrating Docker with Proxmox are:

  1. Install Docker directly on the Proxmox host node.
  2. Install Docker within an LXC container.
  3. Install Docker inside a VM.

Given my need for isolation and security, I opted for the third option: running Docker inside a VM. This approach allows me to benefit from Docker’s deployment simplicity while maintaining the added layer of security that VMs provide. If you’re curious about the performance of these options, I recommend reading this article: Running Docker on Proxmox.

Managing Networking

Proxmox provides its own tools for managing networking, including virtual network interfaces, VLANs, and firewall features. However, since I want more control over my network, I’ve decided to use OpenWRT as my network management solution. I’ll run OpenWRT inside a VM, which will act as the router and firewall for my entire environment.

I considered two other popular options, pfSense and OPNsense, but chose OpenWRT primarily because it is Linux-based. This gives me more flexibility, especially when it comes to managing Wi-Fi networks, as FreeBSD-based systems (like pfSense and OPNsense) have limitations in this area, as you can read from its documentation .

VLANs

The network will be segmented into VLANs to improve security and simplify management. Here’s how I plan to configure them:

  • VLAN 10 (Administration): Reserved for administrative services such as Proxmox and OpenWRT.
  • VLAN 20 (Services): Used to host internal services accessible only to users within the network.
  • VLAN 30 (DMZ): For machines that expose services publicly (e.g., web servers, databases).

Additionally, I will set up two WireGuard VPN networks: one for administration (with access to all VLANs) and one for users (with restricted access to VLAN 20).

Managing Storage

Storage management will be handled with ZFS, which will mirror the two data disks in a RAID-1 configuration. ZFS features like snapshots, encryption, compression, and deduplication will help ensure data reliability and efficiency.

In addition, I’ll configure an LXC container to act as a Network-Attached Storage (NAS) server. This NAS will allow my VMs and other devices to access data over the network using various protocols, providing a customizable and hands-on approach to managing storage.

Managing Identities and Access Control

For identity management, I’ve chosen to use FreeIPA, an open-source alternative to Microsoft Active Directory. To implement Single Sign-On (SSO) for web applications, I’ll use Authentik, which integrates well with FreeIPA.

While Proxmox offers some identity management features, I prefer to have full control over the configuration and use industry-standard tools to build a robust infrastructure.

Organizing the Machines

As previously mentioned, one virtual machine will be dedicated to OpenWRT, acting as router and firewall, and will be responsible for managing the network. An LXC container will serve as the NAS, while another VM will handle user accounts, certificates (PKI), and Single Sign-On (SSO). Another VM will host lightweight applications for user access, ensuring smooth and efficient operations. For heavier applications that manage personal data storage (such as multimedia and documents), a separate VM will be used. Lastly, another VM will be dedicated to monitoring and centralizing logs, ensuring robust security and providing insight into the health of the entire infrastructure.

Conclusion

This post serves as an initial draft of my self-hosting journey, and it’s likely that the setup and decisions outlined here will evolve over time as I continue to implement and refine my infrastructure. Self-hosting is an ongoing learning process, and I expect to modify many aspects of the design as I encounter new challenges and opportunities.

In future blog posts, I’ll dive deeper into the specifics of each phase of the project, providing more detailed insights.
